# Navigation au clavier, ARIA et accessibilité

> **Pourquoi c’est important ?**  
> L’accessibilité permet à chacun d’utiliser un site dans de bonnes conditions, quels que soient ses outils, ses capacités ou sa manière de naviguer.  
> Cela concerne notamment :  
> - les utilisateurs à mobilité réduite ;  
> - les personnes malvoyantes ou aveugles utilisant un lecteur d’écran ;  
> - les personnes qui naviguent uniquement au clavier ;  
> - les utilisateurs ayant des troubles cognitifs ou de l’attention ;  
> - les utilisateurs avancés qui préfèrent les raccourcis clavier pour aller plus vite.  

Une interface accessible ne se limite pas à “fonctionner”.  
Elle doit aussi être **compréhensible**, **prévisible**, **navigable** et **utilisable sans obstacle**.

---

## 1. L’ordre de tabulation : logique et naturel

### 1.1 Suivre l’ordre visuel
Les éléments focusables doivent apparaître dans le même ordre que dans le rendu visuel.

**Exemple :** si le champ “Nom” est au-dessus de “Email” dans le formulaire, il doit aussi être tabulé avant.

Un ordre de focus incohérent désoriente :
- les utilisateurs clavier ;
- les personnes utilisant un lecteur d’écran ;
- les personnes ayant besoin d’une interface prévisible.

### 1.2 Éviter les `tabindex` non nuls
N’utilisez `tabindex="1"` (ou plus) que si vous repensez entièrement l’ordre de navigation, ce qui est rarement souhaitable.

Préférez :
- le comportement natif des éléments HTML ;
- `tabindex="0"` pour rendre focusable un élément qui ne l’est pas naturellement ;
- `tabindex="-1"` lorsqu’un élément doit pouvoir recevoir le focus **par script**, sans entrer dans l’ordre de tabulation.

```html
<!-- Bon exemple : le bouton est naturellement focusable -->
<button>Envoyer</button>

<!-- Possible, mais moins bon qu’un vrai bouton -->
<div role="button" tabindex="0">Lire la suite</div>
````

### 1.3 Bien comprendre `tabindex="-1"`

`tabindex="-1"` ne sert pas à la navigation avec la touche Tab.
En revanche, il peut être très utile pour :

* déplacer le focus vers une modale ;
* amener le focus sur un message d’erreur ;
* envoyer le focus sur une confirmation après une action.

```html
<section id="confirmation" tabindex="-1">
  <h2>Merci</h2>
  <p>Votre formulaire a bien été envoyé.</p>
</section>
```

## 2. Visibilité du focus : guider le regard

Quand un utilisateur navigue au clavier, il doit toujours savoir **où il se trouve**.

### 2.1 Garder un focus visible

Conservez le contour de focus par défaut ou remplacez-le par un style clairement visible.

Évitez absolument :

```css
outline: none;
```

sans alternative visible.

### 2.2 Exemple de style de focus

```css
.button:focus {
  outline: 3px dashed #0066cc;
  outline-offset: 2px;
}
```

Un bon style de focus doit être :

* contrasté ;
* visible sur tous les fonds ;
* cohérent sur l’ensemble du site.

## 3. Composants riches : règles dédiées

Les composants interactifs plus complexes demandent des règles particulières de navigation clavier et parfois l’usage d’ARIA.

| Composant         | Actions clavier                                                                     | Rôles / attributs utiles                                           |
| ----------------- | ----------------------------------------------------------------------------------- | ------------------------------------------------------------------ |
| Menu déroulant    | `Enter` / `Space` pour ouvrir ou fermer, `↑` / `↓` pour naviguer, `Esc` pour fermer | `role="menubar"`, `role="menu"`, `role="menuitem"`                 |
| Onglets           | `Tab` pour atteindre la barre, `←` / `→` pour changer d’onglet                      | `role="tablist"`, `role="tab"`, `role="tabpanel"`, `aria-selected` |
| Dialogue / modale | `Tab` reste dans la modale, `Esc` pour fermer                                       | `role="dialog"` ou `<dialog>`, `aria-modal="true"`                 |

Ces composants doivent être :

* utilisables à la souris ;
* utilisables au clavier ;
* compréhensibles avec un lecteur d’écran ;
* cohérents dans leur comportement.

## 4. ARIA : compléter le HTML, pas le remplacer

ARIA (*Accessible Rich Internet Applications*) permet d’ajouter des informations utiles aux technologies d’assistance.

Mais ARIA ne doit pas servir à remplacer un HTML correct.

### 4.1 Toujours préférer le HTML natif

Quand un élément HTML existe, utilisez-le.

```html
<!-- Meilleur choix -->
<button type="button">Ouvrir le menu</button>

<!-- Moins bon : il faut recréer le comportement clavier et l’accessibilité -->
<div role="button" tabindex="0">Ouvrir le menu</div>
```

Privilégiez en priorité :

* `<button>`
* `<a>`
* `<input>`
* `<select>`
* `<textarea>`
* `<details>`
* `<dialog>`
* les éléments sémantiques comme `<nav>`, `<main>`, `<header>`, `<footer>`

### 4.2 Ce qu’ARIA apporte réellement

ARIA sert surtout à :

* exposer un **état** (`aria-expanded`, `aria-selected`, `aria-invalid`) ;
* exposer une **relation** (`aria-controls`, `aria-describedby`, `aria-labelledby`) ;
* préciser un **rôle** lorsqu’aucun élément natif ne convient ;
* annoncer une **mise à jour dynamique** (`aria-live`).

## 5. Nom accessible : chaque contrôle doit être compréhensible

Tout élément interactif doit avoir un nom accessible clair.

### 5.1 Boutons et liens

Le texte visible suffit souvent :

```html
<button>Envoyer</button>
<a href="/contact">Nous contacter</a>
```

En revanche, un bouton icône seul doit recevoir un nom accessible :

```html
<button aria-label="Fermer">
  <svg aria-hidden="true" focusable="false"><!-- ... --></svg>
</button>
```

### 5.2 Champs de formulaire

Chaque champ doit avoir un label explicite :

```html
<label for="email">Adresse e-mail</label>
<input id="email" name="email" type="email">
```

Pour ajouter une aide complémentaire :

```html
<label for="password">Mot de passe</label>
<input id="password" name="password" type="password" aria-describedby="pwd-help">
<p id="pwd-help">8 caractères minimum.</p>
```

Un champ sans label clair devient difficile, voire impossible, à utiliser avec un lecteur d’écran.

## 6. États et propriétés ARIA les plus utiles

### 6.1 `aria-expanded`

Indique si un contenu dépliable est ouvert ou fermé.

```html
<button aria-expanded="false" aria-controls="faq-1">
  Voir la réponse
</button>

<div id="faq-1" hidden>
  <!-- contenu -->
</div>
```

Quand le panneau s’ouvre :

* `aria-expanded` doit passer à `true` ;
* l’attribut `hidden` doit être retiré.

### 6.2 `aria-current`

Permet d’indiquer l’élément courant, par exemple la page active dans une navigation.

```html
<nav aria-label="Navigation principale">
  <ul>
    <li><a href="/">Accueil</a></li>
    <li><a href="/services" aria-current="page">Services</a></li>
    <li><a href="/contact">Contact</a></li>
  </ul>
</nav>
```

### 6.3 `aria-hidden`

Masque un élément aux technologies d’assistance lorsqu’il est décoratif.

```html
<span aria-hidden="true">★</span>
```

À ne pas utiliser sur :

* un élément interactif ;
* un texte important ;
* une information nécessaire à la compréhension.

### 6.4 `aria-live`

Annonce automatiquement un changement de contenu dynamique.

```html
<p aria-live="polite" id="status"></p>
```

Cas d’usage :

* “Formulaire envoyé” ;
* “Produit ajouté au panier” ;
* résultat mis à jour après une recherche.

`polite` convient dans la majorité des cas.
`assertive` doit rester rare, pour les messages urgents.

## 7. Gestion du focus : accessibilité réelle, pas seulement théorique

Une interface peut sembler accessible visuellement, tout en restant difficile à utiliser au clavier.

### 7.1 Après une action, le focus doit rester logique

Exemples :

* ouverture d’une modale → le focus entre dans la modale ;
* fermeture d’une modale → le focus revient au bouton d’ouverture ;
* soumission d’un formulaire invalide → le focus va au résumé des erreurs ou au premier champ en erreur ;
* affichage d’un message de confirmation → le focus peut être déplacé vers ce message.

### 7.2 Piège fréquent

Un composant “cliquable” à la souris mais impossible à atteindre ou à activer au clavier n’est pas accessible.

Il faut vérifier :

* l’accès au focus ;
* l’activation au clavier ;
* la cohérence du comportement.

## 8. Lien « Aller au contenu »

Les utilisateurs clavier et lecteurs d’écran ont souvent intérêt à éviter de retraverser toute la navigation à chaque page.

Placez un lien d’évitement en haut de page :

```html
<a href="#main-content" class="skip-link">Aller au contenu</a>

<main id="main-content">
  <!-- ... -->
</main>
```

### 8.1 CSS recommandé

```css
.skip-link {
  position: absolute;
  left: -999px;
  top: auto;
  width: 1px;
  height: 1px;
  overflow: hidden;
}

.skip-link:focus {
  left: 0;
  top: 0;
  width: auto;
  height: auto;
  padding: 8px;
  background: #000;
  color: #fff;
  z-index: 1000;
}
```

Le lien reste discret visuellement, mais devient visible au focus.

## 9. Erreurs de formulaire : visibles, explicites, annoncées

Les erreurs doivent être :

* visibles à l’écran ;
* formulées clairement ;
* reliées au champ concerné ;
* détectables par les technologies d’assistance.

```html
<label for="email">Adresse e-mail</label>
<input
  id="email"
  name="email"
  type="email"
  aria-describedby="email-error"
  aria-invalid="true"
>
<p id="email-error">Veuillez entrer une adresse e-mail valide.</p>
```

### 9.1 Bonnes pratiques

* ne pas signaler une erreur uniquement par la couleur ;
* écrire un message compréhensible ;
* relier le message au champ ;
* guider l’utilisateur vers la correction.

Une bordure rouge seule ne suffit pas.

## 10. Landmarks et structure sémantique

L’accessibilité commence souvent par une structure HTML claire.

### 10.1 Grandes zones de page

Les principales zones doivent être identifiables :

* `<header>`
* `<nav>`
* `<main>`
* `<aside>`
* `<footer>`

```html
<header>
  <h1>Mon site</h1>
</header>

<nav aria-label="Navigation principale">
  <!-- liens -->
</nav>

<main id="main-content">
  <!-- contenu principal -->
</main>

<footer>
  <!-- pied de page -->
</footer>
```

### 10.2 Pourquoi c’est utile

Ces landmarks permettent aux utilisateurs de lecteurs d’écran de :

* sauter rapidement d’une zone à l’autre ;
* comprendre la structure générale de la page ;
* repérer plus facilement le contenu principal.

## 11. Tester et valider

### 11.1 Test manuel au clavier

Parcourez le site uniquement au clavier :

* `Tab`
* `Shift + Tab`
* `Enter`
* `Space`
* `Esc`

Vérifiez :

* que tous les éléments interactifs sont atteignables ;
* que l’ordre de navigation est logique ;
* que le focus est toujours visible ;
* que les composants riches réagissent correctement.

### 11.2 Outils automatisés

Quelques outils utiles :

* **Lighthouse** (audit accessibilité) ;
* **axe DevTools / axe-core** ;
* **WAVE** ;
* validateurs HTML.

### 11.3 Lecteurs d’écran

Tester avec un lecteur d’écran donne souvent une autre vision du site :

* **NVDA** sous Windows ;
* **VoiceOver** sur macOS et iPhone ;
* **TalkBack** sur Android.

---

## 12. Pièges fréquents à éviter

* Utiliser ARIA pour compenser un mauvais HTML.
* Transformer des `<div>` ou des `<span>` en faux boutons ou faux liens.
* Supprimer le focus visible sans le remplacer.
* Utiliser `tabindex` positif pour “forcer” l’ordre.
* Oublier de mettre à jour les attributs ARIA quand l’interface change.
* Utiliser `aria-hidden="true"` sur un contenu utile.
* Créer un composant utilisable à la souris mais pas au clavier.
* Donner des consignes visuelles uniquement basées sur la couleur.
* Oublier le lien “Aller au contenu”.
* Faire une modale sans gestion du focus.

---

## 13. À retenir

L’accessibilité ne consiste pas à ajouter “un peu d’ARIA” en fin de projet.

Elle repose d’abord sur :

1. un HTML sémantique ;
2. une navigation clavier correcte ;
3. des libellés clairs ;
4. une gestion cohérente du focus ;
5. des retours compréhensibles ;
6. des composants testés dans des conditions réelles.

ARIA est un complément utile, mais il ne remplace ni la structure, ni la logique, ni les bonnes pratiques de base.
