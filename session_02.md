# Session 2 — Wireframes des pages principales
Durée : **4h**

Cette séance sert à transformer le cadrage du projet en **structure visuelle claire**.  
À partir du travail réalisé lors de la session 1, vous devez représenter les pages principales du site sous forme de **wireframes**.

La théorie, les rappels méthodologiques et les exemples seront fournis dans des fichiers séparés.  
Ici, on se concentre uniquement sur **les résultats à obtenir** à la fin de la séance.

---

# Objectif de la séance

À la fin de cette séance, vous devez disposer d’une **représentation structurée des pages principales** de votre projet.

Le but n’est pas encore de faire un beau design.  
Le but est de décider :

- quelles zones composent chaque page
- dans quel ordre elles apparaissent
- quelles informations elles contiennent
- quels éléments sont communs d’une page à l’autre
- quels éléments relèvent du visiteur ou de l’administration

Un wireframe sert à organiser la page, pas à la décorer.

---

# Résultats attendus en fin de séance

## 1. Les pages principales sont représentées
Vous devez produire les wireframes des pages essentielles du projet.

Selon votre projet, cela inclut généralement au minimum :

- page d’accueil
- page de listing / catalogue
- page de détail
- page de contact ou d’action principale
- page(s) d’administration utile(s), si elles sont déjà identifiées

Toutes les pages n’ont pas besoin d’être détaillées au même niveau, mais les pages principales doivent exister.

---

## 2. La structure de chaque page est claire
Chaque wireframe doit permettre de comprendre :

- où se trouve l’en-tête
- où se trouve la navigation
- où se trouve le contenu principal
- où se trouvent les zones secondaires
- où se trouve le pied de page
- comment l’utilisateur lit et parcourt la page

On attend une structure logique, lisible et cohérente.

---

## 3. Les composants communs sont repérés
Vous devez identifier les éléments qui reviennent sur plusieurs pages.

Par exemple :

- header
- navigation principale
- footer
- cartes de contenu
- boutons d’action
- barre de recherche
- filtres
- formulaire type

Ces éléments récurrents devront ensuite être intégrés proprement en HTML.

---

## 4. La hiérarchie de l’information est visible
Vos wireframes doivent déjà montrer :

- ce qui est prioritaire
- ce qui est secondaire
- ce qui doit être vu immédiatement
- ce qui peut apparaître plus bas dans la page

L’utilisateur doit pouvoir comprendre rapidement la logique de chaque page.

---

## 5. Le parcours utilisateur commence à apparaître
À travers vos wireframes, on doit pouvoir comprendre comment l’utilisateur avance dans le site.

Exemples :

- accueil → listing → détail
- accueil → formulaire de contact
- login admin → tableau de bord → gestion des contenus

Les pages ne doivent pas être pensées isolément, mais comme les étapes d’un parcours.

---

## 6. Le contenu est anticipé sans être finalisé
Un wireframe ne contient pas forcément les textes définitifs, mais il doit déjà prévoir :

- titres
- blocs de texte
- images ou médias
- listes
- boutons
- formulaires
- filtres ou outils de navigation

Il faut donc réfléchir à ce que la page devra accueillir.

---

# Travail à produire pendant la séance

## A. Wireframes des pages principales
Vous devez produire les wireframes des pages essentielles du projet.

Chaque wireframe doit être suffisamment complet pour préparer l’intégration HTML par la suite.

Le niveau de détail attendu doit permettre de savoir :

- quelles zones existent
- à quoi elles servent
- dans quel ordre elles apparaissent

---

## B. Identification des composants récurrents
Vous devez relever les éléments communs entre plusieurs pages.

Le but est de préparer la future logique de composants et de templates.

Cela vous aidera ensuite à éviter les répétitions inutiles dans l’intégration.

---

## C. Vérification de cohérence entre les pages
Vous devez vérifier que vos wireframes forment un ensemble cohérent.

On attend par exemple :

- une logique de navigation stable
- des zones communes placées de manière cohérente
- un parcours utilisateur compréhensible
- une structure compatible avec le contenu prévu

---

# Livrables de fin de séance

À la fin des 4h, vous devez pouvoir montrer :

## Livrable 1 — Wireframe de la page d’accueil
Le wireframe de l’accueil doit montrer clairement :

- les zones principales
- la hiérarchie des contenus
- les accès vers les autres pages importantes

## Livrable 2 — Wireframe de la page de listing / catalogue
Cette page doit permettre de comprendre :

- comment les contenus sont listés
- comment l’utilisateur repère ou filtre ce qu’il cherche
- comment il accède à une fiche détail

## Livrable 3 — Wireframe de la page de détail
Cette page doit permettre de voir :

- comment une information ou un contenu complet est affiché
- quelles actions sont possibles depuis cette page
- quelles informations sont prioritaires

## Livrable 4 — Wireframes complémentaires
Selon le projet, vous devez également prévoir les pages utiles, par exemple :

- contact
- connexion
- tableau de bord
- formulaire d’ajout
- gestion de contenus

## Livrable 5 — Liste des composants communs
Vous devez produire une liste simple des éléments récurrents repérés dans vos pages.

---

# Critères de validation

La séance est considérée comme réussie si :

- les pages principales sont représentées
- la structure des pages est compréhensible
- la hiérarchie de l’information est visible
- les composants communs sont repérés
- le parcours utilisateur est cohérent
- les wireframes sont suffisamment clairs pour préparer l’intégration HTML

---

# Checklist de fin de séance

Avant de quitter la séance, vérifiez que vous avez bien :

- réalisé le wireframe de l’accueil
- réalisé le wireframe du listing
- réalisé le wireframe du détail
- prévu les autres pages nécessaires
- identifié les composants communs
- vérifié la cohérence entre les pages
- réfléchi au parcours utilisateur
- préparé une base exploitable pour la session HTML

---

# Production minimale attendue

Si votre travail est encore incomplet, le minimum indispensable à avoir à la fin de la séance est :

- un wireframe d’accueil
- un wireframe de listing
- un wireframe de détail
- une première liste des composants communs

Sans cela, l’intégration HTML de la séance suivante risque d’être improvisée.

---

# À ne pas faire pendant cette séance

Cette séance n’est pas faite pour :

- produire le design final
- choisir les couleurs définitives
- travailler la typographie finale
- coder en HTML/CSS
- détailler la logique PHP

Le but est de **structurer les pages avant de les intégrer**.
