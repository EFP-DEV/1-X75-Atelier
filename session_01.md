# Session 1 — Cadrage du projet

Durée : **4h**  
Objectif : définir le projet (quoi, pour qui, pourquoi, quelle structure) avec assez de précision pour enchaîner sur les wireframes en session 2.

Vous ne devez pas coder. Vous devez **décider et écrire**.

---

## Livrable 1 — Fiche de cadrage

Un document court, structuré, qui permet à quelqu'un d'extérieur de comprendre votre projet en quelques minutes.

### Contenu attendu

- **Nom du projet**
- **Thème et type de site**
- **Public cible**
- **Objectifs du site**
- **Fonctionnalités principales**
- **Pages principales envisagées**

### Comment bien définir le projet

Votre description doit être précise. Testez-la : si quelqu'un la lit, peut-il comprendre ce que l'utilisateur trouvera sur le site et ce qu'il pourra y faire ?

Évitez les formulations vagues :

- ❌ "un site moderne", "un site pratique", "un site pour tout le monde"

Préférez des formulations exploitables :

- ✅ "site vitrine pour un service de photographie"
- ✅ "catalogue de produits avec fiches détaillées"
- ✅ "site MVC avec espace public et back-office d'administration"

### Comment identifier le public cible

Il ne suffit pas d'écrire "tout public". Répondez à ces questions :

- Qui est l'utilisateur principal ?
- Dans quelle situation arrive-t-il sur le site ?
- Quelle action veut-il accomplir rapidement ?
- Quelles informations lui sont les plus utiles ?

Exemples concrets :

- "un visiteur qui cherche les horaires, les tarifs et un moyen de contact"
- "un client qui parcourt un catalogue puis consulte une fiche détaillée"
- "un administrateur qui ajoute, modifie et supprime des contenus sans toucher au code"

### Comment formuler les objectifs

Les objectifs décrivent ce que le site **doit permettre de faire**, pas ce qu'il doit être. Distinguez les objectifs côté visiteur et côté administration.

Exemples :

- Permettre à un visiteur de consulter des articles ou des produits
- Permettre à un visiteur d'accéder rapidement aux informations essentielles
- Permettre à un administrateur de gérer les contenus sans modifier le code

Limitez-vous à l'essentiel : quelques objectifs clairs valent mieux qu'une longue liste imprécise.

### Comment lister les pages

Produisez une première arborescence simple. Distinguez déjà les pages publiques et les pages d'administration.

Pour vous aider, répondez à :

- Quelles pages sont indispensables pour que le projet ait du sens ?
- Par quelle page l'utilisateur entre-t-il dans le site ?
- Quelle page lui permet de consulter un contenu détaillé ?
- Quelles pages sont réservées à l'administration ?

Exemples de pages courantes : accueil, catalogue/listing, détail, contact, login admin, dashboard, gestion des contenus.

## Livrable 2 — Mini analyse concurrentielle

Observez au moins **3 sites** proches de votre sujet. Le but n'est pas de copier, mais de repérer des solutions utiles et des erreurs à éviter.

### Pour chaque site, notez :

- URL ou nom du site
- Son intérêt par rapport à votre projet
- Un ou deux points réussis
- Un ou deux points faibles
- Ce que vous retenez pour votre propre projet

### Quoi observer ?

- Structure de navigation
- Clarté de la page d'accueil
- Organisation des fiches détaillées
- Lisibilité des menus
- Hiérarchie visuelle
- Facilité à trouver l'information
- Surcharge ou confusion éventuelle

Une référence sans remarque exploitable ne sert à rien. Soyez concrets :

- ✅ "navigation simple et claire, je retiens l'idée d'un listing avec filtres"
- ✅ "page d'accueil trop chargée, j'évite la surcharge visuelle"
- ❌ "site bien fait" (ne dit rien d'utile)

---

## Livrable 3 — Repository initialisé

Le projet doit exister techniquement dès cette séance.

Ce qui est attendu :

- Un repository créé avec un nom clair
- Une arborescence de départ simple
- Un premier commit propre avec un message explicite
- Le dépôt partagé selon les consignes du cours

Pas de fichiers inutiles, pas de commit vide. Le dépôt doit montrer que le travail a commencé.

---

## Livrable 4 — README.md

Le `README.md` est le point d'entrée du projet. Il doit déjà contenir :

- Le nom du projet
- Le thème
- Une courte description
- Le public visé
- Les objectifs principaux

Il doit être cohérent avec la fiche de cadrage. Quelqu'un qui ouvre le repository doit comprendre de quoi il s'agit sans chercher ailleurs.

---

## Ce qu'il ne faut pas faire pendant cette séance

- Travailler la charte graphique, le logo ou les couleurs
- Coder du HTML ou du PHP
- Commencer l'intégration sans savoir quelles pages existent
- Ajouter des fonctionnalités sans objectif défini
- Partir sur un périmètre trop large pour le temps du projet

Le but est de **définir correctement avant de produire**.

---

## Minimum absolu en cas de retard

Si tout n'est pas terminé, vous devez au moins avoir :

- Un sujet clair
- Un public cible identifié
- Une liste de pages
- Un repository créé
- Un README commencé

Sans cela, la session suivante sera difficile à aborder.

---

## Checklist de fin de séance

Avant de quitter, vérifiez que vous avez :

- [ ] Défini le thème et le type de site
- [ ] Identifié le public cible
- [ ] Formulé les objectifs principaux
- [ ] Listé les fonctionnalités principales
- [ ] Listé les pages principales
- [ ] Observé et annoté au moins 3 références
- [ ] Créé le repository
- [ ] Créé le README.md
- [ ] Fait au moins un premier commit

Vérifiez aussi que tout est **rédigé**, pas juste évoqué à l'oral.

---

## Transition vers la session 2

La session suivante part de ce cadrage pour produire les **wireframes** des pages principales.

Le travail réalisé ici doit donc déjà permettre de savoir quelles pages représenter, quel contenu elles contiendront, et quels parcours utilisateur prendre en compte.
